Design Presentation: CacheFlow

Review the proposed architecture and implementation plan

1. Architecture & State Management

  • Structure: Clean Architecture (Data, Domain, Presentation).
  • State: flutter_bloc for predictable state transitions (Scanning -> Idle -> Cleaning -> Done).
  • DI: get_it and injectable for decoupled dependencies.

2. Native Bridge & Scanning

  • Kotlin MethodChannels: Will expose StorageStatsManager to get cache/data/APK sizes.
  • Performance: The list parsing and sorting will be offloaded to a Dart Isolate (via the compute function) to keep the UI smooth if there are 100+ apps.

3. Core Cleaning Engine

  • Auto-Detect Strategy: The app will silently check for `su` access. If granted, it runs shell commands. If not, it requests Accessibility permissions.
  • Accessibility Flow: A Kotlin AccessibilityService will be built to loop through the "Top Offenders" list, open Settings, click "Storage", and click "Clear Cache".

4. Storage & UI

  • Database: Hive will be used to store historical data (e.g., "Total MB cleaned this week").
  • Dashboard: Material 3 design, featuring a bar chart breakdown of storage and a highlighted section for the "Top 3 Offenders".
